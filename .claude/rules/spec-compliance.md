## 공식 스펙 준수 원칙 (최우선)

**이 프로젝트의 Mock 서버는 테스트 편의를 위한 가짜 서버가 아니다.
실제 PG 클라이언트 코드가 코드 변경 없이 그대로 동작해야 하는 호환 서버다.**

### 절대 규칙

1. **PG 공식 스펙에 없는 기능을 임의로 추가하지 마라.**
   - 공식 API에 없는 헤더, 파라미터, 응답 필드를 Mock에 넣으면 안 된다.
   - 예: 나이스 공식 스펙에 Idempotency-Key가 없으면 mock-nice에도 구현하지 않는다.
   - "있으면 편하겠다"는 이유로 기능을 넣는 순간, 실무 코드가 Mock에서만 동작하는 코드를 작성하게 된다.

2. **Endpoint, HTTP method, 요청/응답 구조는 공식 스펙과 100% 동일하게 유지하라.**
   - URL path, 파라미터명, 필드명, 상태값, 에러코드는 공식 문서 그대로.
   - 토스는 `code`/`message`, 나이스는 `resultCode`/`resultMsg` — 통일하지 마라.

3. **인증 방식은 공식 스펙을 따르라.**
   - 토스: `Basic {Base64(secretKey:)}`
   - 나이스: `Basic {Base64(clientKey:secretKey)}`
   - 편의상 통일하거나 간소화하지 않는다.

4. **에러 응답 포맷은 PG 공식 포맷 그대로 반환하라.**
   - Mock 전용 래퍼(`ApiResponse`, `ErrorWrapper` 등)로 감싸지 않는다.
   - 실제 PG 클라이언트의 에러 파싱 로직이 그대로 동작해야 한다.

### Mock에서만 허용되는 것

아래는 Mock의 특성상 공식과 다를 수밖에 없는 항목이다. 이것만 허용한다.

| 허용 항목 | 이유 |
|-----------|------|
| 결제창/인증 과정 생략 | Mock은 서버→서버 API만 제공. 결제창 UI 없음. |
| 에러 트리거 (키워드 매칭) | 특정 에러를 확정적으로 발생시키기 위한 Mock 전용 메커니즘 |
| 카오스 모드 (`/chaos/*`) | 장애 시뮬레이션을 위한 Mock 전용 엔드포인트 |
| 카드/은행 정보 고정값 | 실제 카드사 연동 없으므로 고정값 사용 |
| `X-CHAOS-MODE` 헤더 | 개별 요청 카오스 오버라이드용 Mock 전용 헤더 |

### 판단 기준

새 기능을 추가하기 전에 반드시 확인:

1. **해당 PG의 공식 문서에 이 기능이 있는가?**
   - 있다 → 구현
   - 없다 → 구현하지 않음
2. **`docs/{pg}/spec/`에 근거가 있는가?**
   - 스펙 문서에 없는 것은 코드에도 없어야 한다.
3. **"실무 클라이언트가 이 기능에 의존하는가?"가 아니라, "실제 PG가 이 기능을 제공하는가?"로 판단하라.**
